# Целевая UX/UI-спецификация AI Discovery Platform

Дата: 2026-05-17

## Цель UX

Платформа должна быть рабочим инструментом Product Owner и бизнес-аналитика, а не демонстрационной landing-страницей. Основной экран должен помогать быстро понять состояние проекта, качество контекста, готовность этапов, вопросы AI и следующий безопасный шаг.

## Пользовательский путь

1. Создать проект Discovery.
2. Заполнить ручной контекст и загрузить источники.
3. Дождаться извлечения текста и анализа readiness.
4. Ответить на уточняющие вопросы AI.
5. Сгенерировать проблему, цель и бизнес-эффект.
6. Уточнить AS IS, TO BE и Use Cases.
7. Сформировать функциональные и нефункциональные требования.
8. Проверить полноту и риски.
9. Собрать финальный БТ.
10. Экспортировать DOCX и передать в delivery.

## Информационная архитектура

- Главная: список проектов, статусы, дата изменения, готовность БТ.
- Проект: dashboard с прогрессом, текущими блокировками, последними AI-запусками.
- Контекст: источники, ручное описание, extraction status, chunks, readiness, source trace.
- Этапы Discovery: Проблема, Цель, Бизнес-эффект, AS IS, TO BE, Use Cases, Требования, Риски, Финальный БТ.
- Настройки: LLM provider, корпоративный режим, проверка подключения.
- Экспорт: DOCX, validation report, приложения с источниками.

## Требования к UI Kit

- Единые кнопки primary/secondary/ghost/danger.
- Единые badges для статусов: черновик, готово, предупреждение, ошибка, заблокировано.
- Единые empty/loading/error states.
- Единые панели AI-действий и AI-вопросов.
- Единая таблица source trace с источником, chunk, score, версией и использованием в артефакте.
- Компоненты должны быть плотными, сканируемыми и пригодными для регулярной работы.

## Экран «Контекст»

Обязательные блоки:
- Ручное описание инициативы.
- Документы и ссылки.
- Статусы extraction: загружен, готов, пустой, неподдерживаемый, ошибка.
- Knowledge coverage: процессы, системы, роли, KPI, ограничения, риски.
- Readiness score и причины снижения.
- Missing information.
- Source trace и список chunks.
- Действие «Обновить контекст».

## Этапы Discovery

Для каждого этапа нужны:
- Заголовок этапа и краткий статус.
- Основной редактор или структурированная форма.
- AI action bar.
- AI questions с ответами пользователя.
- Блок source trace.
- История версий.
- Состояние зависимости: актуально, устарело, заблокировано.

## Визуализация readiness и progress

- Readiness отображается как процент, статус и список причин.
- Progress отображается по этапам, а не только общей полосой.
- Заблокированные этапы объясняют причину и действие для разблокировки.
- Устаревший артефакт показывает источник изменения, например обновление Context или Goal.

## UX-состояния

- Пусто: объяснить, что нужно заполнить, без длинных инструкций.
- Загрузка: показывать действие, которое выполняется.
- Ошибка: дать понятное сообщение и повторяемое действие.
- Предупреждение: разрешить продолжение, но показать риск.
- Заблокировано: не скрывать этап, а объяснить dependency.
- Готово: показать версию, дату и источник результата.

## Accessibility basics

- Контраст текста не ниже WCAG AA.
- Все интерактивные элементы доступны с клавиатуры.
- Ошибки не передаются только цветом.
- Таблицы имеют явные заголовки.
- Длинные labels не должны ломать layout.
